# Cotomo Dev/Ops Standard

> **「ルールブックではなく、判断基準のフレームワーク」**
>
> このドキュメントは、200万ユーザのCotomoを開発・運用するエンジニア組織の行動原則と実践基準を定めるものである。
> 状況は常に変わる。だからこそ「なぜそうするのか」の哲学を共有し、各自が状況に応じて判断できる組織を目指す。

## 1. Philosophy — 我々が大切にすること

### 3つの柱

#### Ownership — 「自分ごと」として捉える

- 自分が書いたコードだけでなく、自分が関わるプロダクト全体に対するOwnershipを持つ
- 「それは○○さんの担当です」は禁句。気づいた人が声を上げ、動く
- 自分が作ったものが本番でどう動いているかを知ることが、Ownershipの原点

#### 学習と透明性 — 失敗を隠さない

- 障害報告、ポストモーテム、技術的な判断の過程をすべてドキュメント化し公開する
- 「知らなかった」は罪ではない。「知ろうとしなかった」が問題
- コードレビューは「指摘の場」ではなく「学習の場」。レビューコメントには「なぜ」を必ず書く

#### Pragmatic Idealism — 現実主義的な理想主義

- 理想的なアーキテクチャを知った上で、今の制約の中で最善の判断をする
- 「正しい」ことと「今やるべきこと」は違う。技術的に美しい解が常にベストとは限らない
- ただし、妥協する時は意識的に妥協し、必ず記録を残す（ADR: Architecture Decision Records の活用）

### 根底にある考え方

- **スピードと品質は二項対立ではない。** 本当に速いチームは品質も高い。品質が低いとリワークが発生し、結果的に遅くなる
- **障害ゼロは目指さない。ただし、同じ障害を二度起こさない。** 障害ゼロを目指す組織はリスクを取れなくなり、イノベーションが止まる
- **仕組みで解決し、個人の頑張りに依存しない。** 個人の善意や努力に頼る組織は、キーパーソンが抜けた瞬間に崩壊する
- **Output（作ったもの）ではなく Outcome（生み出した価値）に向き合う。** リリースして終わりではない

## 2. Delivery — 価値を届ける

### リリースとデプロイの分離

- **デプロイ = コードを本番環境に配置すること**（技術的な行為）
- **リリース = ユーザに機能を公開すること**（ビジネス的な判断）
- Feature Flag を活用し、デプロイのリスクを下げ、ビジネス側がリリースタイミングをコントロールできる状態を作る

### 「Done」の定義

PRがマージされた ≠ 完了。本番環境でユーザに価値が届き、メトリクスで確認できた時点が「Done」。

### デプロイ戦略

**Canaryデプロイを標準とする**（200万ユーザ規模では必須）

| Step | トラフィック比率 | 観察時間 |
|------|----------------|---------|
| 1 | 1% | 5分 |
| 2 | 10% | 10分 |
| 3 | 50% | 15分 |
| 4 | 100% | — |

- 各ステップでエラー率・レイテンシを自動チェック。閾値超過で自動ロールバック
- ホットフィックスでもCanaryプロセスは省略しない。急いでいる時こそ事故る

### ロールバック

- ロールバックは「前のバージョンのコンテナイメージに戻す」だけで完了する設計にする
- **1コマンド、1分以内**で完了できること
- DB migrationは必ず前方互換にする（カラム追加 → コード変更 → カラム削除の3ステップ）
- 判断基準: エラー率が通常の2倍、またはp99レイテンシが通常の3倍を超えたら即実行

### 開発プロセス

- **1週間スプリント**を推奨。小さく出して素早く学ぶサイクルが命
- **WIP制限は「チーム人数 - 1」を上限**。WIPが増えるとコンテキストスイッチで全員の生産性が落ちる
- PRレビュー待ちはWIPにカウントする
- **レビュー待ち24時間ルール**: PRが24時間以上レビュー待ちなら、朝会で優先的に対応
- Trunk-based developmentを推奨。長寿命ブランチは負債そのもの

### Outcomeへの向き合い方

- **Feature Success Criteria をリリース前に定義**する。「この機能をリリースすると○○の指標が△△改善する」
- リリース2週間後に判定。成功していなければ改善するか撤退するかを判断する
- Discoveryフェーズにエンジニアを巻き込む。「仕様が決まったから作って」ではOutcomeに向き合えない

## 3. Reliability — 信頼性を守る

### SLO / SLI / エラーバジェット

| 用語 | 定義 | Cotomoでの例 |
|------|------|-------------|
| **SLI** (Service Level Indicator) | ユーザ体験に直結する計測指標 | API応答時間のP99、エラー率、会話の接続成功率 |
| **SLO** (Service Level Objective) | SLIに対する目標値 | 月間可用性 99.9%（≒ ダウンタイム約43分/月） |
| **SLA** (Service Level Agreement) | ユーザとの契約上の約束 | SLOより緩い値で設定 |
| **エラーバジェット** | SLOを満たしつつ許容される障害量 | 月間43分のダウンタイム枠 |

- **エラーバジェットが「スピードと品質」のバランスを定量化する唯一の方法**
- バジェットを使い切ったら、安定化に集中する。バジェットに余裕がある限り、積極的にリリースする
- 99.9% が現実的な出発点。99.99% はコストが跳ね上がるため、事業フェーズに合わせて判断

### グレースフルデグラデーション

200万ユーザのサービスでは「全部止まる」が最悪のシナリオ。機能ごとの重要度を定義し、段階的に落とす設計にする。

| Tier | 定義 | 例 |
|------|------|-----|
| **Tier 1** | 絶対止めない | 音声会話のコア機能 |
| **Tier 2** | 短時間なら許容 | 履歴参照、設定変更 |
| **Tier 3** | 一時的に停止可 | 分析、レコメンド、通知 |

- バックエンドサービスがダウンした場合のフォールバック値を事前に定義する
- タイムアウト設定を全外部呼び出しに入れる。デフォルト値の放置は障害を長期化させる
- サーキットブレーカーで障害の連鎖を防止する

## 4. Observability — 見える化する

### 基本方針: 「ユーザより先に異常に気づく」

### 3本柱

| 柱 | 目的 | 基準 |
|---|---|---|
| **Metrics** | 「何が起きているか」を数値で把握 | RED (Rate/Errors/Duration) + USE (Utilization/Saturation/Errors) |
| **Logs** | 「なぜ起きたか」を追跡 | 構造化ログ（JSON）必須、リクエストID付与 |
| **Traces** | 「どこで遅いか」をリクエスト単位で追跡 | OpenTelemetry準拠 |

### 必ず計測するメトリクス

**システムメトリクス**
- Rate: リクエスト数/秒（エンドポイント別）
- Errors: エラー率（4xx/5xx別、エンドポイント別）
- Duration: レイテンシ（p50, p95, p99）。**平均は嘘をつくので使わない**
- Utilization: CPU、メモリ、ディスク、コネクションプール

**ビジネスメトリクス**
- DAU/WAU、セッション継続時間、会話ターン数、音声認識成功率
- これらの急変はシステム障害の先行指標になることが多い

### アラート設計

- アラートは「人間が今すぐアクションを取るべきもの」だけに絞る
- **SLOベースのアラート**: エラーバジェットの消費速度で発火
- **アラート疲れ（Alert Fatigue）は最大の敵**。月次でアラートのS/N比をレビューし、不要なアラートを積極的に削除する
- 月に1回以上「対応不要」で閉じるアラートは削除 or 修正する
- 全アラートに対応するランブックを用意し、アラートからランブックへの直リンクを貼る

### ログレベルの定義

| レベル | 用途 |
|--------|------|
| **ERROR** | ユーザ影響あり、即対応が必要 |
| **WARN** | 潜在的問題、近いうちに対応が必要 |
| **INFO** | 正常な業務イベント（リクエスト開始/完了等） |
| **DEBUG** | 本番では原則OFF |

- PII（個人情報）はログに出力しない。マスキングを強制する仕組みを入れる

## 5. Incident Management — 障害に向き合う

### Severity 定義

| レベル | 定義 | 対応基準 |
|--------|------|---------|
| **Sev1** | 全ユーザ or 主要機能停止 | 即時対応、全員招集可能 |
| **Sev2** | 一部機能・一部ユーザに影響 | 30分以内に対応開始 |
| **Sev3** | 軽微な不具合 | 次営業日対応可 |

**迷ったら上げる。** 上げすぎて怒られることはない、上げなくて大事になることはある。

### インシデントコマンダー（IC）制度

- ローテーション制のICを導入。ICは「対応する人」ではなく「指揮する人」
- 自分でコードを触らず、状況把握・判断・コミュニケーションに集中する
- ICローテーションは週次。シニアだけでなくミドルレベルも含める

**ICチェックリスト**
1. 影響範囲の特定
2. ステークホルダーへの初報（15分以内）
3. 対応チームの招集
4. 定期的なステータスアップデート
5. 解決確認
6. ポストモーテムのスケジュール

### エスカレーションフロー

- 15分で解決しなければエスカレーション。一人で抱え込ませない
- Slackに `#incident` チャンネルを設け、障害対応はすべてそこで実施。DMやスレッドで情報が分散しない

### ポストモーテム

**Blameless（個人を責めない）は絶対条件。人ではなくシステムを改善する。**

- Sev1/Sev2は必須。障害発生から1週間以内に完了する
- 当事者ではない第三者がファシリテートする
- 「なぜあなたはこうしたのか」ではなく「この時点でシステムはどういう状態だったか」と問う

**ポストモーテムの構成**
1. タイムライン（検知 → 対応 → 復旧 → 完了）
2. 影響範囲とユーザインパクト
3. 根本原因
4. なぜ検知が遅れたか
5. うまくいったこと / 改善すべきこと
6. アクションアイテム（担当者・期限付き）

- **アクションアイテムは次のスプリントプランニングに組み込む。** 「次から気をつける」は改善ではない
- 月次の「障害共有会」で過去事例をカジュアルに共有する

### ランブック

- 全アラートに対応するランブックを用意する
- 必須項目: ①影響範囲の確認方法 ②一次対応手順 ③根本原因の調査手順 ④エスカレーション基準と連絡先
- 障害対応後にすぐ更新する。古いランブックは害悪

## 6. Engineering Practices — 技術基盤

### CI/CD パイプライン

**「mainは常にデプロイ可能」**

| 基準 | 目標 |
|------|------|
| CI全体 | 15分以内（hard limit） |
| ビルド時間 | 5分以内 |
| デプロイ時間 | 15分以内 |
| ビルド成功率 | 95%以上 |
| デプロイ成功率 | 99%以上 |

- PRごとに: lint → unit test → integration test → セキュリティスキャン
- コンテナイメージのタグはgit commit SHA。`latest`タグは本番環境では使わない
- flaky testは即修正 or 隔離。CI全体への信頼が崩壊する

### コードレビュー

**「レビューはボトルネックではなくセーフティネット」**

- PRは変更300行以下を目安に小さく保つ
- レビュー開始までのSLA: 4時間以内（営業時間内）
- 必須: 1名以上のApprove。セキュリティ関連・DB migration・インフラ変更は2名以上
- CIが通っていないPRはレビューしない

**レビューで見るべきポイント（優先度順）**
1. 正しさ（バグ、エッジケース）
2. セキュリティ（入力バリデーション、認証認可、データ露出）
3. パフォーマンス（N+1、不要なメモリ確保）
4. 可読性（6ヶ月後の自分が読めるか）

スタイル・命名は lint に任せる。人間がスタイルで議論するのは時間の無駄。

### マージ戦略

- Squash Mergeを基本とする
- mainへの直接pushは禁止（branch protection必須）
- マージ後のブランチは自動削除

### Feature Flag 運用

- 大きな機能変更はflag経由でmainにマージ
- 命名規約: `ff_{feature_name}_{YYYYMMDD}`
- **導入から90日以内に削除する。** 放置flagは技術的負債
- 状態管理はコードではなく外部設定で行う

### テスト戦略

**「テストは品質を証明するものではなく、変更を安全にするもの」**

```
        /   E2E   \        ← 少数、クリティカルパスのみ
       / Integration \      ← API境界、DB接続、外部連携
      /   Unit Tests   \    ← 大量、高速、ビジネスロジック中心
```

| 種別 | 比率 | 速度基準 |
|------|------|---------|
| Unit | 70% | 1テスト1秒以内、全体2分以内 |
| Integration | 20% | 実DBコンテナ使用 |
| E2E | 10% | クリティカルパスのみ |

- カバレッジ目標: ビジネスロジック層80%以上。ただし数値を目的化しない
- 負荷テスト: 通常ピークの2倍のトラフィックに耐えることを確認
- 本番環境でのSynthetic Monitoring（合成監視）も導入

### インフラ管理

- IaC（Terraform/Pulumi）必須。手作業でのインフラ変更は禁止
- インフラ変更もPRベースでレビュー
- 環境間（dev/staging/prod）の差分を最小化

### 技術的負債

- **負債をバックログに可視化し、ビジネスインパクトで優先度をつける。** 「いつかやる」リストにしない
- 通常のバックログに統合し、技術タスクとビジネスタスクを同じ土俵で優先順位付けする
- **ボーイスカウトルール**: 触ったコードは、触った時より少しだけ良くして戻す
- 四半期に1度の Tech Health Review でアーキテクチャ全体を見直す
- 妥協した判断はADR（Architecture Decision Records）に記録する

**リソース配分の目安**

| 区分 | 比率 |
|------|------|
| 新機能開発 | 60% |
| 技術基盤整備・負債返済 | 25% |
| 探索的取り組み（PoC等） | 15% |

※ エラーバジェットの消費状況やビジネス要件に応じて柔軟に調整する

## 7. Security — 安全を組み込む

### 基本方針: 「セキュリティはリリース前のゲートではなく、開発プロセスに組み込む」

### CIパイプラインに組み込むセキュリティチェック

| チェック | 目的 | タイミング |
|----------|------|-----------|
| SAST（静的解析） | コードの脆弱性検出 | PRごと |
| SCA（依存関係スキャン） | 脆弱性のある依存の検出 | PRごと + 定期 |
| コンテナスキャン | ベースイメージの脆弱性 | ビルド時 |
| シークレット検出 | APIキー等のコミット防止 | PRごと |

- Critical/Highの脆弱性は72時間以内に対応

### アプリケーションセキュリティ

- 認証・認可のロジックは共通ライブラリ化し、各サービスで独自実装しない
- APIは認証必須をデフォルトにし、公開エンドポイントを明示的にopt-inする
- ユーザデータの暗号化: 保存時（at rest）と通信時（in transit）の両方
- PII（個人情報）はログに出力しない

### 依存関係管理

- 自動更新（Renovate / Dependabot）を有効化
- マイナー/パッチは自動マージ可。メジャーバージョンアップは月次で評価
- 使われていない依存は定期的に削除

## 8. Team & Culture — 組織と文化

### 組織設計

**「You Build It, You Run It」を段階的に導入する**

| Phase | 開発チームの責任 | SRE/Platformの役割 |
|-------|----------------|-------------------|
| Phase 1 | デプロイまで責任を持つ | 運用の一次対応 |
| Phase 2 | 自チームのオンコールに参加 | プラットフォーム・ツール提供 |
| Phase 3 | End-to-Endオーナーシップ | イネーブリングチーム |

- 1チームは5〜8名。ドメイン別に3〜5チーム
- 各チームにSRE的スキルを持つメンバーを最低1名配置（Embedded SRE）
- 横断的なPlatform Engineeringチームが共通基盤を提供
- サービスオーナーシップマトリクスを明文化する

### オンコール制度

| 項目 | 基準 |
|------|------|
| ローテーション人数 | 最低4〜5名 |
| 体制 | Primary / Secondary の2名体制 |
| シフト期間 | 最長1週間 |
| 週末含む場合 | 代休必須 |
| エスカレーション | 15分で解決しなければ上位へ |

- **オンコール手当は必須。** 拘束に対する金銭的補償なしに持続可能な制度は作れない
- 呼ばれなかった場合でも拘束手当は支払う
- 負担の偏りを月次で可視化し、偏りがあればマネージャーが介入して是正する
- **自動復旧の推進**: 人が起きなくても直るものは自動化する

### 心理的安全性

- **Blamelessは「制度」ではなく「実践」。** ドキュメントに書くだけでは不十分
- 経営層・マネジメント層が自ら「自分のミス」を共有する。上が見本を見せなければ下は安心しない
- 「バカな質問」をしても大丈夫な空気を意識的に作る
- 「わからない」と言える仕組み: レビューコメントの意図がわからない時に聞きやすい文化
- コードレビューでは「なぜこうするのか」を説明するコメントを意識する

### 評価とインセンティブ

**評価に組み込む観点**
- 信頼性への貢献（SLO達成率、トイル削減の実績）
- 障害対応の質（迅速さ、ポストモーテムの質、再発防止策の実行度）
- 知識共有（Runbook整備、ドキュメント作成、他チームへの技術支援）

**絶対にやってはいけないこと**
- 障害の発生件数で評価すること → 「何もデプロイしない」が最適解になる
- 運用タスクを「雑務」として評価から除外すること

- 「運用は罰ゲームではない」というメッセージをマネジメントが繰り返し発信する
- 運用改善に取り組んだエンジニアを可視化し称賛する
- SRE/Platform方向へのキャリアパスを開発と同等に位置づける

### 持続可能なペース

- スプリントキャパシティの20%は技術的負債返済・改善・学習に充てる。これは交渉ではなく前提条件
- 残業が2週間連続で発生したらスコープを削る。「頑張れば間に合う」は持続不可能のサイン
- 四半期ごとにチームの休暇取得率をチェック

## 9. Growth — 成長する組織

### オンボーディング

**段階的な責任委譲**

| 期間 | 経験すること |
|------|-------------|
| **1〜3ヶ月目** | ペアでの開発、小さなバグ修正、テスト追加、本番への小さなデプロイ |
| **4〜6ヶ月目** | 小機能の単独開発、コードレビュー参加、オンコールシャドウイング |
| **7〜12ヶ月目** | 中規模機能のリード、ICサブ担当、ポストモーテム執筆 |

- 入社1ヶ月目にプロダクションへの小さなデプロイを経験させる
- 障害対応のシャドウイング制度を設ける（見学 → 副担当 → 主担当）
- **ペアデプロイ制度**: 最初の数回は先輩と一緒にデプロイする
- **Good First Issue ラベルを常に5〜10個ストック**しておく

### ドキュメント文化

- **「新人はまずこれを読め」リストを整備する**
- 暗黙知を形式知にする。Slackの過去ログに重要な意思決定を埋もれさせない
- 重要な意思決定はドキュメントに残す（ADR）
- オンボーディングドキュメントを一箇所に集約する。断片的に散らばらせない
- 開発環境のセットアップ手順をメンテし、自動化する
- **ドキュメントはコードの隣に置く。** 外部ドキュメントは陳腐化する

### 用語集

新人が「えっと…」とならないように、以下を含む用語・概念のドキュメントを整備する:

- SLO / SLI / SLA / エラーバジェットの定義とCotomoでの具体的な値
- カナリアリリース、ブルーグリーンデプロイの定義と自社での採用方式
- Feature Flag の運用ルール
- Severityレベルの定義と判断基準
- ポストモーテムの文化と実際の運用方法

### 学習機会

- **Game Day**: 四半期に1回、意図的に障害を起こしチームの対応力を鍛える
- **社内Tech Talk**: 月1回、障害事例やナレッジの共有
- **過去障害のケーススタディ研修**: 新メンバー向け
- **週1回の1on1**: 技術的な話だけでなく、不安や困りごとも聞く
- **ペアプログラミング/モブプログラミング**: 週2回以上

### スケーラビリティへの備え

- チーム間の依存関係を疎にするアーキテクチャを意識する（APIコントラクト、イベント駆動設計）
- エンジニアリングマネージャー層の計画的な育成
- 採用基準にDev/Opsマインドセットを含める

## 10. Metrics — 計測と改善

### DORA Metrics（Four Keys）

| メトリクス | Elite水準 | 最初の現実的目標 |
|-----------|----------|----------------|
| デプロイ頻度 | 1日複数回 | 週3回以上 |
| リードタイム（コミット→本番） | 1時間未満 | 1日以内 |
| 変更失敗率 | 0-15% | 15%以下 |
| MTTR（復旧時間） | 1時間未満 | 4時間以内 |

### 追加メトリクス

- **PR サイクルタイム**: 作成 → マージまでの時間
- **デプロイあたりの変更量**: 大きすぎるデプロイはリスク
- **オンコール対応回数・深夜対応回数**: 負担の可視化
- **チーム満足度**: 四半期アンケートで定点観測

### 計測の原則

- **メトリクスは「チーム」の単位で見る。個人のパフォーマンス評価には絶対に使わない**
- 最初から完璧な計測を目指さない。取れるものから始める
- 絶対値よりトレンド（変化の方向）が重要
- **計測なくして改善なし**

### 導入のアプローチ

全部を一度にやろうとすると失敗する。**四半期ごとに2〜3個の施策に絞って導入し、定着したら次に進む。**
