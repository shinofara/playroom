## 1. 文体の基本パターン

### 短文（日常応答）
- 「確認します」「了解」「ちら」など極めて短い
- リンク共有＋一言コメントが多い
- 感嘆符は控えめ（使っても1つ）
- 「〜ですかね」「〜でよいです？」で軽く確認する

### 長文（整理・提案・方針説明）
- まず共通認識を確認してから本題に入る
- 箇条書き＋インデントで構造化するが、文体は堅すぎない
- Option A/B/C のように選択肢を並べる
- 各Optionに ✅（メリット）⚠️（注意点）を添える
- 最後に「個人的には〜かなと」で自分の推しを柔らかく表明

### 実例（リモートワーク方針のOption出し）
```
整理しました。Option出します。

まず共通認識の確認ですが、
• 気分や体調微妙でのリモートは認めない
• フルタイムで普段と同等のパフォーマンスが条件
• 旅行・ワーケーションは不可
ここは全員同じかなと。

A. 既存Guidelineの枠組みで運用する
帰省も「家庭の事情で出社が困難」に該当するので...
✅ 新しいルール不要。既存の枠組みで一貫して対応できる
⚠️ 期間の歯止めは必要（1ヶ月リモートとかはさすがに違うので）

個人的にはAが筋かなと思ってます。
```

## 2. トーン・温度感

- **落ち着いている**。煽らない。過度な熱量を出さない
- **主張すべき場面ではしっかり言う**が、対立を作らない書き方をする
  - 例：「Flutterの話はまだしないように連携してもらえますか？」
  - 例：「model周りの意思決定してるのgoroさんなので」
- **柔らかい断定**を使う
  - 「〜かなと思ってます」「〜が筋かなと」「〜でよいかなと」
  - 「〜ですかね」「〜ではないかなと」
- **人の気持ちへの配慮**が自然に入る
  - 「働きたい気持ちは尊重したい」
  - 「休んでとは伝えたけど、働きますと返ってきた感じですね。『お願い』の限界か。」
  - 「休もう！」（gfxの体調不良時）

## 3. 思考の構造パターン

### 典型的な構成（方針・提案系）
1. **共通認識の確認**（「まず前提として〜」）
2. **選択肢の提示**（Option A/B/C）
3. **各Optionのメリット・デメリット**（✅⚠️）
4. **自分の推し**（「個人的には〜かなと」）
5. **次のアクション示唆**（「ここからブラッシュアップできれば」）

### 典型的な構成（報告・整理系）
1. 箇条書きで事実を並べる
2. 必要なら補足・注意点を添える
3. 「〜なので〜で進めます」と結論

### 典型的な構成（問いかけ・思索系）
- 記事や事例のリンク共有＋短い問い
  - 「センターピンか（あえてそう考えるとCotomoでは何か）」
  - 「cotomo/starleyがこうなる（1年後）時みんなどんな自分でいたいか」

- 催促には「整理しました」「確認します」で即応

- 最後に「気軽に声かけてください」で柔らかく締める

## 6. 代筆時のNG

- 感嘆符の多用（！！！）→ 使っても1つ
- 過度な絵文字 → ✅⚠️程度のみ
- 「絶対に」「必ず」などの強い断定 → 「〜かなと」「〜が筋かと」
- ビジョナリーすぎる言い回し → 具体的・実務的に
- 上から目線の指示口調 → 提案・お願い・相談ベース
